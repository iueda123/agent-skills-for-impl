# Single-Tree Implementation Workflow

`/impl-single` は、git worktree を増やさず現在の作業ツリー1つで実装を進めるためのスキル。
大きく並列化せず、1タスクずつ丁寧に進めたいときに使う。

## 想定ユースケース

- 「今回は並列でなく丁寧に進めたい」
- 「単一ブランチで実装・検証・修正を回したい」
- 「変更範囲を小さく保ちながら確実に進めたい」

## 基本方針

1. 作業前に現状確認（ブランチ、未コミット差分、依存関係）
2. 変更を小さく分割し、1ステップごとに確認
3. 実装のたびにテスト/静的チェックを実行
4. 不明点は放置せず、その場で確認してから進める
5. 最後に変更要約と残課題を明示

## ワークフロー

### Step 0: コンテキスト状態を確認

実装開始前にコンテキストウィンドウ使用率を確認する。
50%以上なら、実装に入る前に必ず次の趣旨で compact を促す。

```text
コンテキストウィンドウ使用率が50%以上です。作業を安定して継続するため、先に compact を実行してください。
```

### Step 1: 作業コンテキストを確認

最初に以下を確認する。

```bash
git branch --show-current
git status --short
```

必要なら依存関係やセットアップ手順も確認する（例: `README.md`, `package.json`, `build.gradle`）。

### Step 2: 実装計画を小さく分割

実装を一度に進めず、2〜5個程度の小さな作業単位に分割する。
各単位は「変更対象」「期待結果」「確認方法」をセットで定義する。

### Step 3: 1単位ずつ実装・検証

1単位ごとに次を繰り返す。

1. 必要最小限の変更を入れる
2. テストまたはビルド/静的チェックを実行する
3. 結果を確認し、次の1単位に進む

不明点・曖昧点がある場合は推測で進めず、ユーザー確認またはコード調査を行う。

### Step 4: 仕上げ

最終的に以下をまとめる。

- 何を変更したか
- どの確認を実施したか
- 残っているリスク/未対応項目
- 次に行うべき最小ステップ

### Step 5: 完了まで一気通貫で進める

タスクとフェーズがすべて完了するまで途中で止まらず進める。
進捗が追えるよう、plan ファイルに ToDo チェックボックスがある場合は、完了した項目に都度チェックを入れる。

## ルール

- worktree を前提にしない。現在の単一ツリーで進める
- 大きな変更を一気に入れず、差分を小さく保つ
- テスト/チェックなしで完了扱いにしない
- 不確実な仕様は推測で確定しない
- ユーザーが「じっくり丁寧に」を求める場合、速度より検証の確実性を優先する
- 実装開始前にコンテキストウィンドウ使用率が50%以上なら、compact を必ず促す
- すべてのタスクとフェーズが完了するまで作業を継続し、plan の完了ToDoへ都度チェックを入れる

## 前提条件

- git リポジトリ内で作業していること
- 現在の作業ツリーに書き込み権限があること
- プロジェクトのテスト/ビルドコマンドが利用可能であること
