# Refine Plan

create-plan で作られた plan ファイルを対象に、Claude が対話しながら内容整理・追記・修正を支援する。

## ワークフロー

### Step 1: 改善対象のplanを決める

最初に、必ず「どの plan ファイルを改善するか」を確認する。

対象が未指定なら、`docs/prompts/**/*.md` を列挙して候補を提示し、ユーザーに選んでもらう。
候補が見つからない場合は、ユーザーに対象ファイルのパスを指定してもらう。

確認メッセージは次の趣旨で行う。

```text
どの plan を改善しますか？
対象ファイルのパスを指定してください（例: docs/prompts/project-a.md）。
```

### Step 2: plan本文を読み、改善観点を整理する

対象ファイルを読んで、少なくとも次を確認する。

- 目的とスコープが明確か
- Wave/依存関係の並びが妥当か
- 各Issue向け実行手順が具体的か
- 前提条件・制約・完了条件が不足していないか
- 参照ドキュメントや既存コード参照が不足していないか

### Step 3: 対話で改善する

対象ファイルの改善を対話で進める。
必要に応じて、本文の構造化・表現の統一・不足情報の追記を提案し、ファイルへ反映する。
相談・依頼内容が小規模なものとなってきたら、曖昧な点や不明な点がなくなるまで、ユーザーに1問ずつ確認して進める。

編集支援開始時は次の趣旨で尋ねる。

```text
どのような観点で改善しますか？
依存関係の見直し、実行手順の具体化、スコープ整理、リスクの明確化、文章の整形などを一緒に進められます。
```

区切りで次の趣旨を確認する。

```text
この内容で plan ファイルに反映しましょうか？
```

### Step 4: ToDoリストを補完する

plan ファイルに実装完了までの ToDo リストがない場合は、チェックボックス付きで追記する。

追記例:

```markdown
## ToDo
- [ ] 要件の最終確認
- [ ] 実装
- [ ] テスト
- [ ] ドキュメント更新
- [ ] レビュー対応
```

## ルール

- create-plan の生成意図（並列実装の実行可能性向上）を損なわない
- plan に書かれていない前提を勝手に確定しない。推測は推測として明示する
- 相談・依頼内容が小規模な段階では、曖昧点が解消するまで1問1答で確認する
- 実装完了までの ToDo リストがない場合は、チェックボックス形式で追記する
- 「注釈に対応して」と依頼された場合、注釈はユーザーがAIに文書への追記・修正を依頼するためのものとして扱う
- 「注釈に対応して」と依頼された場合、回答は注釈を使わず本文へ統合した形で提示する
- 大幅なスコープ変更が必要な場合は、plan 修正だけでなく関連 Issue の見直しも提案する

## 前提条件

- 現在のディレクトリが対象リポジトリ
- create-plan で生成した plan ファイル（例: `docs/prompts/*.md`）にアクセスできる
