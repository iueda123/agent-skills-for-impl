## 5. SKILL.md のワークフロー設計

### Step 1: 対象 PR の特定

- 現在のブランチから `gh pr view` で PR 情報を取得
- PR 本文の `Closes #xxx` / `close #xxx` / `Fix #xxx` 等から Issue 番号を抽出

### Step 2: 対象 plan ファイルの特定

- スキル呼び出し時の引数から plan ファイルパスを取得
- 引数がない場合: ユーザーに尋ねる（`AskUserQuestion` を使用）

  ```
  振り返り対象の plan ファイルパスを教えてください（例: docs/plans/xxx.md）
  指定なしの場合はスキップします。
  ```

### Step 3: 整合性チェック — Issue DoD vs 実装コード

- `gh issue view {番号}` で DoD（完了条件チェックリスト）を取得
- `git diff develop..HEAD` で変更ファイルを確認
- 各 DoD 項目に対して「対応する変更が確認できるか」を評価

### Step 4: 整合性チェック — plan ファイル vs 実装コード

- 指定された plan ファイルを読む
- 計画の「実施ステップ」「変更対象」「完了条件」と実装差分を照合

### Step 5: テスト実行（Gradle）

- `./gradlew test` を実行
- 結果（PASS/FAIL、テスト数）を記録
- テスト失敗時は報告書に失敗詳細を含める

### Step 6: 報告書生成

- `references/report-template.md` を使って報告書を生成
- コメント冒頭に識別子 `<!-- tidy-up-report -->` を付与

### Step 7: PR コメント投稿

- `gh pr comment {PR番号} --body "..."` で投稿

### Step 8: ローカル保存

- `gh-pr-log/{PR番号}/comments.md` に保存
- 冒頭に以下を付記:

  ```
  この内容は PR コメントとして GitHub 上に記録済みのため破棄しても構わない。
  ```

## 7. トレードオフと採用理由

| 選択肢 | 採用理由 |
|---|---|
| Markdown スキルとして実装 | 既存スキルと同一パターン。追加ツール不要 |
| `<!-- tidy-up-report -->` 識別子 | 既存コメントとの区別、将来的な更新検出が容易 |
| `gh-pr-log/` ローカル保存 | PR コメント削除後も参照可能。履歴として蓄積しやすい |
| Gradle 固定（初版） | Issue の想定対象が明示されている。他ランナーは後続 Issue で対応 |

## 8. リスクと回避策

| リスク | 回避策 |
|---|---|
| PR 本文に `Closes #xxx` がない | 抽出できなかった場合はユーザーに Issue 番号を尋ねる |
| テストが長時間かかる | タイムアウト（例: 5分）を設定し、超過時は旨を記録 |
| `./gradlew` が存在しない | ファイル存在チェックし、ない場合はテストをスキップ扱いにする |
| plan ファイルの構造が想定外 | 構造不明時は「計画との照合スキップ」として報告書に記録 |
| PR コメント投稿失敗 | エラーを報告し、ローカル保存は必ず実行する |
