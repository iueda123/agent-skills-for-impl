# tidy-up Report Template

報告書生成時はこのテンプレートを埋めて出力する。
`{...}` は実値に置換する。不要なブロックはコメントで示した条件に従い削除する。

---

```markdown
<!-- tidy-up-report -->
# tidy-up レポート

PR: #{number}
実行日時: {datetime}
ブランチ: {branch}
plan ファイル: `{plan_file_path}`

---

## Axis 1: Issue DoD チェック

Issue: #{issue_number} {issue_title}

| # | DoD 項目 | 状態 | 備考 |
|---|----------|------|------|
| 1 | {dod_item} | ✅ 実装済み / ❌ 未実装 / ❓ 不明 | {note} |

---

## Axis 2: plan ファイル整合性チェック

| ステップ | 状態 | 備考 |
|---------|------|------|
| {step} | ✅ 完了 / ❌ 未完了 / ➖ スコープ外 | {note} |

---

## Axis 3: テスト結果 / 整合性評価

プロジェクト種別: {project_type}

<!-- Java/Gradle の場合（project_type が "Java/Gradle" のときのみ出力） -->
```
{gradle_test_output_summary}
```
- 合計: {total} 件 / 成功: {passed} 件 / 失敗: {failed} 件

<!-- テストなし（R/Python/Bash/Markdown等）の場合（上記以外のときのみ出力） -->
| 確認項目 | 状態 | 備考 |
|---------|------|------|
| {item} | ✅ 一致 / ❌ 逸脱 / ❓ 不明 | {note} |

---

## 総合評価

{overall_assessment}
```

---

## 記入ルール

### {project_type} の値

| 条件 | 値 |
|------|----|
| `build.gradle` が存在する | `Java/Gradle` |
| `*.R` ファイルが主体 | `R スクリプト` |
| `*.py` ファイルが主体 | `Python スクリプト` |
| `*.sh` ファイルが主体 | `Bash スクリプト` |
| `*.md` ファイルが主体 | `Markdown 文書` |
| 判定不能 | ユーザーに確認して記入 |

### 状態アイコンの使い方

- ✅ : 実装/完了/一致が確認できた
- ❌ : 未実装/未完了/逸脱が確認できた
- ❓ : 差分から判断できない（テストがない、コードが見つからないなど）
- ➖ : 計画書に記載があるが今回の PR スコープ外

### Axis 3 整合性評価モードの確認観点（テストなしプロジェクト向け）

以下の観点を参考に確認項目を列挙すること。

- 想定していた出力ファイル・図表・スクリプトが存在するか
- 使用データ・手法・出力形式が Issue/Plan の記述と一致するか
- Markdown 文書の場合: 章・節の構成が plan の設計通りか
- スクリプトの場合: 主要な処理ステップ（読み込み・変換・出力）が意図通りか

### {overall_assessment} の書き方

1〜3文で総合判断を記述する。例:

- "3軸すべて問題なし。PR は DoD を満たしており、マージ可能と判断します。"
- "Axis 1 の DoD 項目 #3 が未実装。マージ前に対応が必要です。"
- "Axis 3 で出力ファイルが1件未確認。確認後にマージを推奨します。"
