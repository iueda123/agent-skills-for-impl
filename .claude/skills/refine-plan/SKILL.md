# Refine Plan

create-plan で作られた Issue 向け計画ファイルを対象に、Claude が対話しながら内容整理・追記・修正を支援する。
このスキルは計画フェーズ専用であり、実装指示は扱わない。

## ワークフロー

### Step 1: 改善対象のIssue計画ファイルを決める

最初に、必ず「どの Issue 計画ファイルを改善するか」を確認する。

対象が未指定なら、次の順で候補を提示する。

1. `docs/plans/*/issues/*.md` を列挙し、候補を提示してユーザーに選んでもらう
2. 候補がない場合のみ、ユーザーに対象ファイルのパスを指定してもらう

確認メッセージは次の趣旨で行う。

```text
どの Issue 計画ファイルを改善しますか？
対象ファイルのパスを指定してください（例: docs/plans/project-a/issues/1-setup.md）。
```

### Step 2: Issue計画本文を読み、改善観点を整理する

対象ファイルを読んで、少なくとも次を確認する。

- Issue の目的とスコープが明確か
- 依存関係（前提Issue/後続Issue）の並びが妥当か
- 実行手順と完了条件が具体的か
- 前提条件・制約・完了条件が不足していないか
- 参照ドキュメントや既存コード参照が不足していないか
- 「まだ実装しない」が文書全体で一貫しているか

### Step 3: 対話で改善する

対象ファイルの改善を対話で進める。
必要に応じて、本文の構造化・表現の統一・不足情報の追記を提案し、ファイルへ反映する。
相談・依頼内容が小規模なものとなってきたら、曖昧な点や不明な点がなくなるまで、ユーザーに1問ずつ確認して進める。

編集支援開始時は次の趣旨で尋ねる。

```text
どのような観点で改善しますか？
依存関係の見直し、計画手順の具体化、スコープ整理、リスクの明確化、文章の整形などを一緒に進められます。
```

区切りで次の趣旨を確認する。

```text
この内容で Issue 計画ファイルに反映しましょうか？
```

### Step 4: ToDoリストを補完する

Issue 計画ファイルに計画完了までの ToDo リストがない場合は、チェックボックス付きで追記する。

追記例:

```markdown
## ToDo
- [ ] 要件の最終確認
- [ ] 依存関係（Wave）の確定
- [ ] Issueごとの計画書作成
- [ ] 注釈反映・修正
- [ ] 不明点の質問整理
- [ ] 実装フェーズ移行可否の判断
```

## ルール

- create-plan の生成意図（計画作成の実行可能性向上）を損なわない
- plan に書かれていない前提を勝手に確定しない。推測は推測として明示する
- 相談・依頼内容が小規模な段階では、曖昧点が解消するまで1問1答で確認する
- 計画完了までの ToDo リストがない場合は、チェックボックス形式で追記する
- 実装・テスト・PR作成の指示は追加しない
- 必要に応じて、後続フェーズとして `/impl-single` または `/impl-multi` を参照するに留める
- 「注釈に対応して」と依頼された場合、注釈はユーザーがAIに文書への追記・修正を依頼するためのものとして扱う
- 「注釈に対応して」と依頼された場合、回答は注釈を使わず本文へ統合した形で提示する
- 大幅なスコープ変更が必要な場合は、plan 修正だけでなく関連 Issue の見直しも提案する
- plan の前提やスコープが大きく変わった場合は、Wave 構成（直列/並列）・実行順序・Issue の並び順を再評価し、変更案を明示してユーザー合意後に反映する

## 前提条件

- 現在のディレクトリが対象リポジトリ
- create-plan で生成した Issue 計画ファイル（例: `docs/plans/{plan-name}/issues/{issue_number}-{slug}.md`）にアクセスできる
